## Anti-Patterns Verdict

**LLM assessment**: Not blatant AI slop. The interface reads as a credible student/portfolio engineering tool, but it has generated-UI fingerprints: header feature pills, repeated 01/02 section markers, uppercase eyelines, soft-shadow cards everywhere, and a generic placeholder badge. The product model is strong, but the surface still feels more like a styled assignment than an engineering instrument.

**Deterministic scan**: The bundled detector reported 2 warnings in `index.html:9`: `overused-font` for Google Fonts Inter and `single-font` because Inter is the only font used. In product-register context, these are likely acceptable or low-priority false positives because one well-tuned sans is often correct for tools.

**Visual overlays**: No reliable user-visible overlay is available. Browser runtime failed before mutable injection could be tested. Fallback evidence came from a successful headless Chrome render and DOM load.

## Overall Impression

The core direction is right: a restrained engineering workspace with inputs, diagram, results, equations, and honest solver limits. The biggest opportunity is making the first interaction trustworthy: placeholders should not look like entered values, unsupported support choices should be framed before they block calculation, and the mobile layout needs to stop clipping the working surface.

## What's Working

- The engineering model is visible: supports, beam, loads, reactions, equations, and solver limits are present in one workspace.
- Native form controls make the tool familiar and preserve basic keyboard accessibility.
- The fixed-support limitation is handled honestly instead of pretending the old RA/RB formulas still apply.

## Priority Issues

### [P1] Mobile viewport clipping

**Why it matters**: On a 390px viewport, controls, diagram headers, result headings, and notes are clipped. This makes the primary task feel broken for mobile users and portfolio reviewers on phones.

**Fix**: Remove the mobile `margin-right`, audit width constraints, and make diagram/result areas intentionally scroll or reflow. The diagram can have horizontal scroll, but cards and form controls should not clip.

**Suggested command**: `$impeccable adapt index.html`

### [P1] Placeholder values look like real input

**Why it matters**: `10.00` and `2.50` look like valid entered values while the interface says the point load is missing. That contradiction breaks trust at the first interaction.

**Fix**: Either make those actual default values or remove numeric placeholders and add a distinct “Load example” action.

**Suggested command**: `$impeccable clarify index.html`

### [P2] Diagram is too passive before calculation

**Why it matters**: The diagram is the largest surface, but initial state is mostly a beam on a grid. For a learning tool, the largest surface should teach the model.

**Fix**: Add a valid default example, ghosted preview loads, or field-linked annotations that update while users type.

**Suggested command**: `$impeccable shape diagram preview`

### [P2] Solver limits are discoverable too late

**Why it matters**: Fixed support is presented as an equal option, then users discover calculation is disabled only after selection. That makes the tool feel incomplete rather than intentionally scoped.

**Fix**: Add support compatibility copy beside the selectors and label fixed supports as visual-only or “coming next” until the solver supports fixed-end reactions.

**Suggested command**: `$impeccable clarify support selectors`

### [P2] Accessibility status feedback is under-specified

**Why it matters**: Validation, preview, solved state, and result changes are visual updates but not clearly live-announced. Keyboard and screen-reader users may miss state changes.

**Fix**: Add `aria-live` to validation/results/status regions, ensure visible focus states, and update the SVG accessible label when diagram contents change.

**Suggested command**: `$impeccable harden index.html`

## Persona Red Flags

**Alex (Power User)**: No presets, no auto-calculate, no keyboard shortcut for Calculate/Reset, and no quick way to compare load cases. The workflow is credible but slow.

**Jordan (First-Timer)**: P, RA, RB, UDL, and x_bar assume prior knowledge. The placeholder/error mismatch makes the first step feel contradictory.

**Sam (Accessibility-Dependent User)**: Status changes are not live-announced, button focus treatment is weak, SVG accessible text is static, and mobile clipping blocks readable content.

**Project-specific student user**: A civil engineering student can see equations but does not get a clear explanation of how point load, UDL, and triangular load change resultant location.

**Project-specific portfolio reviewer**: The reviewer sees domain effort, but the mobile clipping and placeholder contradiction could make the project feel less finished than the code actually is.

## Minor Observations

- Topbar pills feel more like SaaS tags than engineering controls.
- The `S` placeholder mark has no clear meaning.
- `Sum MA = 0` and `x_bar` would feel more polished with more conventional notation or explanation.
- The dashed placeholder card is visually heavier than the empty state deserves.
- The background grid plus card shadows adds more visual texture than the task needs.

## Questions to Consider

- Should the diagram become the primary input surface, not only the output?
- Are the example numbers meant to teach? If yes, why are they not actual defaults?
- Should Fixed Support be selectable equally, or explicitly labeled as diagram-only until the solver supports it?
- What should a portfolio reviewer remember after 10 seconds: the UI styling, the structural model, or the interaction quality?
